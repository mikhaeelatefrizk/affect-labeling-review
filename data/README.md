# `data/` — datasets accompanying the systematic review

This directory contains every machine-readable dataset produced or used by the review. Everything in here is released under [CC-BY-4.0](../LICENSE-DATA).

Datasets fall into three layers:

| Layer | What it is | Authority |
|-------|------------|-----------|
| **Canonical** | Values exactly as published in the manuscript / preregistration. Restructured for machine-readability but not re-derived. | Source of truth |
| **Derived** | Independently regenerated by re-running the pre-registered query against PubMed today, or by parsing canonical sources into structured form. Useful for ML training and reproducibility checks. | Auxiliary |
| **Reference** | Schemas, codebooks, and search-strategy documentation. | Documentation |

## Note on `included_papers.csv` and the 22-vs-100 distinction

The canonical PRISMA flow records 100 included papers at the synthesis stage. `data/screening/included_papers.csv` contains only the **22 of 100** that can be programmatically and unambiguously identified from the analysis code (`meta-analysis/run_meta_analysis.py` and `supplementary/build_rob_figure.py`). The remaining 78 are cited in the manuscript prose without per-citation inclusion labels, so the boundary between "included study" and "background citation" cannot be cleanly reconstructed.

To partially close this gap, `data/screening/all_references.csv` provides every entry from `references.bib` as a structured row, flagged for confirmed-include status, with a `manuscript_mentions` count column that you can use as a soft inclusion signal (high counts ⇒ probably included).

Full discussion: [`data/screening/README.md`](screening/README.md) §"Why 22 and not 100", and [`docs/faq.md`](../docs/faq.md).

## Data dictionary

For every CSV in this repository, columns and their semantics are documented in the closest `README.md` (each subfolder has one). Schemas live next to the data they describe.

For machine-readable validation, every CSV is checked against either:

1. A **JSON Schema** (under `data/screening/`), or
2. A **frictionless data package descriptor** (the build job `make validate` runs `frictionless validate` on the whole tree).

If a column isn't documented, that's a bug — file a `data` issue.

## Re-running the derivation

```bash
python scripts/extract_included_list.py    # → data/screening/included_papers.csv
python scripts/build_all_references.py     # → data/screening/all_references.csv
make derive                                # → derived_corpus.csv + derived_screening_log.csv
make validate
make quality
```

All scripts are deterministic given a fixed PubMed snapshot. Because PubMed grows over time, hit counts will drift; see `QUALITY_REPORT.md` for the difference between today's re-query and the canonical 1,842 records reported in the manuscript.
