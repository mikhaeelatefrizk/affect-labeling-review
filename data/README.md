# `data/` — datasets accompanying the systematic review

This directory contains every machine-readable dataset produced or used by the review. Everything in here is released under [CC-BY-4.0](../LICENSE-DATA).

Datasets fall into three layers:

| Layer | What it is | Authority |
|-------|------------|-----------|
| **Canonical** | Values exactly as published in the manuscript / preregistration. Restructured for machine-readability but not re-derived. | Source of truth |
| **Derived** | Independently regenerated by re-running the pre-registered query against PubMed today. Useful for ML training and reproducibility checks. | Auxiliary |
| **Reference** | Schemas, codebooks, and search-strategy documentation. | Documentation |

## File index

```
data/
├── README.md                                 ← this file
├── searches/
│   ├── README.md                             ← what's in this subfolder
│   ├── search_strategy.md                    ← PRISMA-S compliant report (canonical)
│   └── search_strategy.csv                   ← structured database × query × date table
├── screening/
│   ├── README.md                             ← screening log documentation
│   ├── screening_log.schema.json             ← JSON Schema for the screening log
│   ├── screening_log.template.csv            ← empty template + 1 example row
│   ├── included_papers.csv                   ← canonical 100 included papers (Phase 2)
│   ├── derived_corpus.csv                    ← derived: PubMed re-query output (Phase 2)
│   └── derived_screening_log.csv             ← derived: corpus joined with included list (Phase 2)
├── exclusion_reason_codebook.md              ← 6 full-text exclusion-reason codes
└── QUALITY_REPORT.md                         ← automated quality report (Phase 2)
```

Files marked **(Phase 2)** are produced by `make derive`. They will appear after a full build but are *not* hand-authored.

## Data dictionary

For every CSV in this repository, columns and their semantics are documented in the closest `README.md` (each subfolder has one). Schemas live next to the data they describe.

For machine-readable validation, every CSV is checked against either:

1. A **JSON Schema** (under `data/screening/`), or
2. A **frictionless data package descriptor** (the build job `make validate` runs `frictionless validate` on the whole tree).

If a column isn't documented, that's a bug — file a `data` issue.

## Re-running the derivation

```bash
make derive
make validate
make quality
```

The derivation is deterministic given a fixed PubMed snapshot. Because PubMed grows over time, hit counts will drift; see `QUALITY_REPORT.md` for the difference between today's re-query and the canonical 1,842 records reported in the manuscript.
