# `data/screening/` — the screening corpus

This directory contains everything related to which papers were retrieved, screened, and ultimately included in the review.

## What's here

| File | Layer | Description |
|------|-------|-------------|
| `screening_log.schema.json` | Reference | JSON Schema 2020-12 specifying the structure of any screening log file. |
| `screening_log.template.csv` | Reference | Empty template with header row + 1 illustrative example row. Drop your screening tool's export here, conformed to this schema, and PRISMA outputs will rebuild automatically. |
| `included_papers.csv` | Canonical | The 100 included papers, with subset assignment, extracted from `references.bib` and `manuscript/manuscript.md`. Generated by `scripts/extract_included_list.py`. |
| `derived_corpus.csv` | Derived | The corpus of candidate records re-retrieved from PubMed using the canonical PROSPERO query. Generated by `scripts/build_derived_corpus.py`. |
| `derived_screening_log.csv` | Derived | `derived_corpus.csv` joined against `included_papers.csv` to produce binary include/exclude labels. Generated by `scripts/build_derived_screening_log.py`. **This is the file most relevant to ML researchers.** |

## Important: what this dataset is and isn't

### What it is

A reproducible, schema-validated corpus of approximately 1,800 candidate papers retrieved from PubMed using the canonical pre-registered search query, each labeled with a binary `included_in_synthesis` flag. Suitable for training and evaluating AI-assisted SLR screening systems.

### What it is not

The original per-paper screening decisions made during the systematic review were not preserved in a shareable form. Specifically, the dataset *does not* contain:

1. **Per-paper exclusion reasons.** Aggregate counts for the 182 full-text exclusions (broken out into 6 reasons) are in `prisma/prisma_counts.txt`. The 1,289 title/abstract exclusions were not categorized per-paper.
2. **The title/abstract vs. full-text decision distinction.** In the derived dataset, `included_in_synthesis` collapses both stages into a single binary label.
3. **The exact original corpus.** PubMed is a living database; re-running the pre-registered query today will retrieve more papers than the original 1,842 (since publications continue to be indexed). The drift is reported in `data/QUALITY_REPORT.md`. The canonical 1,842 number from the published PRISMA flow remains authoritative for the manuscript.

These three limitations are common to retrospective derivation of SLR data. Please report them honestly when re-using the dataset.

## Schema

See [`screening_log.schema.json`](screening_log.schema.json) for the full JSON Schema. Required fields: `record_id`, `title`, `decision`. All other fields are optional. The schema accepts both:

- Canonical screening logs (with explicit per-paper decisions and exclusion reasons), and
- Derived screening logs (with only binary inclusion labels).

## Validation

```bash
python scripts/validate_screening_log.py
```

This will:
1. Schema-validate every screening log CSV against the JSON Schema.
2. Cross-check that the count of `included` rows matches the canonical 100 from `prisma_counts.txt`.
3. Cross-check that subset assignments sum to the canonical 42 + 9 + 28 + 12 + 9 = 100.

Failures are emitted with row-level error messages.

## Methodological note

This review was screened by a single coder. Inter-rater agreement statistics are not available. If you re-screen the corpus with an independent coder, please publish the second screening pass as a separate, dated CSV (e.g., `screening_log__second_screen_2026.csv`) rather than overwriting the existing data. Cohen's κ between the two passes can then be computed reproducibly.
